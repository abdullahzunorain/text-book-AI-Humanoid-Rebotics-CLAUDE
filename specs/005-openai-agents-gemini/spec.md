# Feature Specification: OpenAI Agents SDK with Gemini Models

**Feature Branch**: `005-openai-agents-gemini`  
**Created**: 2026-03-06  
**Status**: Draft  
**Input**: User description: "pls use openAI sdk for agents and for LLM models to integrate in these agents use the gemini model... use the gemini LLM compatable with open AI sdk(chat.completion model)... MUST dont use AGENTIC SDK of google"

## Summary

Replace the existing hand-rolled LLM client (chain-of-responsibility failover across Gemini/Groq/OpenAI) with the **OpenAI Agents SDK** (`openai-agents`). Each major backend capability — RAG chatbot, content personalization, Urdu translation — becomes a dedicated **Agent** with its own instructions, tools, and guardrails. Gemini models are accessed exclusively through Google's **OpenAI-compatible endpoint** (`generativelanguage.googleapis.com/v1beta/openai/`) using the standard `chat.completions` interface. **No Google agentic SDK** (`google-genai`, ADK, Vertex Agent Builder) is used for generation or orchestration.

## Constraints & Non-Goals

- **MUST NOT** use Google's Agent Development Kit (ADK), Vertex AI Agent Builder, or any Google-proprietary agentic framework.
- **MUST NOT** use `google-genai` SDK for LLM generation (it may remain solely for embeddings via `embed_content`, which has no OpenAI-compatible equivalent on Gemini).
- **MUST** use OpenAI Agents SDK (`openai-agents` package) for all agent orchestration.
- **MUST** use Gemini models via the OpenAI-compatible `chat.completions` endpoint (`generativelanguage.googleapis.com/v1beta/openai/`).
- **Non-goal**: Changing the frontend UI — the same API contracts are preserved.
- **Non-goal**: Replacing the vector database (Qdrant) or embeddings pipeline.
- **Non-goal**: Multi-agent conversations or inter-agent delegation in v1 (single-agent-per-request pattern).

## Assumptions

- **Gemini OpenAI-compatible endpoint** supports `chat.completions` with `gemini-2.5-flash` (or equivalent model name mapped by Google).
- The `openai-agents` SDK allows configuring a custom `base_url` and `api_key` to point at Gemini's OpenAI-compatible endpoint instead of OpenAI's own servers.
- Embeddings will continue using `google-genai` SDK's `embed_content()` because Gemini's OpenAI-compatible layer does not yet expose an embeddings endpoint.
- Existing Qdrant collection, database schema, and migrations remain unchanged.
- Existing test patterns (pytest + pytest-asyncio, httpx TestClient) remain the same.
- The failover chain (Gemini → Groq → OpenAI) is replaced by a single Gemini-via-OpenAI-SDK path; if more resilience is needed later, the Agents SDK's model configuration can be extended.

## User Scenarios & Testing

### User Story 1 — Agent SDK Infrastructure & Configuration (Priority: P1)

The development team configures the OpenAI Agents SDK to use Gemini models through Google's OpenAI-compatible endpoint, replacing the existing hand-rolled `LLMClient`. All agents share a common model configuration that points at Gemini.

**Why this priority**: Infrastructure must exist before any individual agent can work.

**Independent Test**: Import the agent configuration module, create a test agent, and verify it can complete a simple `chat.completions` call through the Gemini endpoint.

**Acceptance Scenarios**:

1. **Given** `GOOGLE_API_KEY` is set in environment, **When** the agent infrastructure initializes, **Then** an OpenAI-compatible client pointing at `generativelanguage.googleapis.com/v1beta/openai/` is created.
2. **Given** the agent infrastructure is initialized, **When** a simple prompt is sent through a test agent, **Then** a valid response is returned from Gemini.
3. **Given** `GOOGLE_API_KEY` is missing, **When** the backend starts, **Then** a clear error is logged and agent-dependent endpoints return 503.

### User Story 2 — RAG Chatbot Agent (Priority: P1)

A student opens the textbook website, types a question in the chatbot widget, and receives an accurate answer grounded in textbook content — powered by a dedicated **Tutor Agent** running through the OpenAI Agents SDK with Gemini as the underlying model.

**Why this priority**: The chatbot is the primary user-facing feature; it must work end-to-end before any other agent.

**Independent Test**: Send `POST /api/chat` with a question about ROS 2 and verify the response includes a relevant answer and source references, with the backend using the Agents SDK (not the old `LLMClient`).

**Acceptance Scenarios**:

1. **Given** the backend is running with OpenAI Agents SDK configured for Gemini, **When** a user sends a question via `POST /api/chat`, **Then** the system returns an answer generated by the Tutor Agent with relevant textbook sources.
2. **Given** an authenticated user sends a question, **When** the Tutor Agent produces an answer, **Then** the question and answer are saved to chat history.
3. **Given** Gemini's OpenAI-compatible endpoint is temporarily unavailable, **When** a user asks a question, **Then** the system returns a 503 with a user-friendly error message.
4. **Given** the user highlights textbook text and asks a question, **When** the selected text is sent as context, **Then** the Tutor Agent scopes its answer to that passage.

### User Story 3 — Personalization Agent (Priority: P2)

An authenticated user clicks "Personalize" on a chapter page. A dedicated **Personalization Agent** adapts the chapter prose to the user's learning background (Python level, robotics experience, math level) while preserving code blocks unchanged.

**Why this priority**: Personalization is the second most-used feature and benefits from structured agent instructions that enforce code-block preservation.

**Independent Test**: Send `POST /api/personalize` with a valid JWT and chapter slug; verify personalized content is returned and code blocks are intact.

**Acceptance Scenarios**:

1. **Given** an authenticated user with a "beginner" profile, **When** they request personalization for a chapter, **Then** the Personalization Agent returns simplified explanations with code blocks unchanged.
2. **Given** a personalization result already exists in cache, **When** the same user requests the same chapter, **Then** the cached result is returned without calling the agent.
3. **Given** the user updates their learning background, **When** they next request personalization, **Then** stale cache is invalidated and the agent generates fresh content.

### User Story 4 — Translation Agent (Priority: P2)

An authenticated user clicks "اردو میں پڑھیں" on a chapter page. A dedicated **Translation Agent** translates the chapter prose to Urdu while preserving all code blocks, markdown formatting, and technical terms in English.

**Why this priority**: Translation has identical architectural needs to personalization (agent + cache + code-block preservation) and can be built in parallel.

**Independent Test**: Send `POST /api/translate` with a valid JWT and chapter slug; verify Urdu content is returned with code blocks intact and technical terms preserved.

**Acceptance Scenarios**:

1. **Given** an authenticated user, **When** they request translation for a chapter, **Then** the Translation Agent returns Urdu prose with code blocks and technical terms preserved.
2. **Given** a translation result exists in cache, **When** the same chapter is requested again, **Then** the cached result is returned.
3. **Given** the Gemini endpoint is rate-limited, **When** a translation is requested, **Then** the system returns 429 with a Retry-After header.

### User Story 5 — Backward-Compatible API & Existing Tests (Priority: P1)

All existing API contracts (`POST /api/chat`, `POST /api/translate`, `POST /api/personalize`, `GET /api/chat/history`) remain unchanged. Frontend code requires zero modifications. Existing tests pass with updated mocks.

**Why this priority**: Backward compatibility is mandatory — no frontend changes allowed.

**Independent Test**: Run the full test suite (`python -m pytest tests/ -v`) and verify all 119+ tests pass after the migration.

**Acceptance Scenarios**:

1. **Given** the agent migration is complete, **When** the full test suite runs, **Then** all existing tests pass (with updated mocks where needed).
2. **Given** the frontend sends requests in the existing format, **When** the backend processes them, **Then** responses match the existing API contracts exactly.
3. **Given** the old `LLMClient` is removed, **When** code imports are checked, **Then** no references to the old failover chain remain in production code.

### Edge Cases

- What happens when Gemini's OpenAI-compatible endpoint returns a model name mismatch or unsupported parameter? → Agent configuration must use only parameters supported by Gemini's compatibility layer.
- What happens when a very long chapter exceeds Gemini's context window? → The system must handle token-limit errors gracefully with a user-friendly message, same as today.
- What happens when the OpenAI Agents SDK has a breaking version update? → Pin the SDK version in `requirements.txt`.
- What happens when `google-genai` is used for embeddings but the API key is only set as `GOOGLE_API_KEY`? → Embedding client must read from `GOOGLE_API_KEY` (already the case).

## Requirements

### Functional Requirements

**Agent Infrastructure**

- **FR-001**: System MUST use the `openai-agents` Python package for all agent orchestration.
- **FR-002**: System MUST configure agents to use Gemini models via Google's OpenAI-compatible endpoint at `https://generativelanguage.googleapis.com/v1beta/openai/`.
- **FR-003**: System MUST authenticate to the Gemini endpoint using the `GOOGLE_API_KEY` environment variable.
- **FR-004**: System MUST NOT use any Google agentic SDK (ADK, Vertex Agent Builder, `google-genai` generation methods) for LLM generation or orchestration.
- **FR-005**: System MUST keep `google-genai` SDK only for the embeddings pipeline (`embed_content()`).
- **FR-006**: System MUST define each agent with explicit instructions (system prompt), model name, and optional tools.

**Tutor Agent (RAG Chatbot)**

- **FR-007**: System MUST create a Tutor Agent with the existing RAG system prompt focused on Physical AI textbook content.
- **FR-008**: Tutor Agent MUST receive retrieved Qdrant context as part of its input.
- **FR-009**: Tutor Agent MUST support optional `selected_text` context for scoped answers.
- **FR-010**: System MUST save chat messages to history for authenticated users after the agent responds.
- **FR-011**: Tutor Agent MUST stay on topic and redirect off-topic questions.

**Personalization Agent**

- **FR-012**: System MUST create a Personalization Agent with instructions to adapt content to learner profiles.
- **FR-013**: Personalization Agent MUST receive the user's background profile (Python level, robotics experience, math level, hardware access, learning goal).
- **FR-014**: Personalization Agent MUST preserve all code blocks unchanged in its output.
- **FR-015**: System MUST check cache before invoking the agent and store results in cache after.
- **FR-016**: System MUST invalidate personalization cache when the user updates their background.

**Translation Agent**

- **FR-017**: System MUST create a Translation Agent with instructions for English→Urdu translation.
- **FR-018**: Translation Agent MUST preserve all code blocks, markdown formatting, and technical terms in English.
- **FR-019**: Translation Agent MUST use formal Urdu (Nastaliq script).
- **FR-020**: System MUST check cache before invoking the agent and store results in cache after.
- **FR-021**: System MUST enforce rate limiting (10 requests/minute per IP) on the translate endpoint.

**API Compatibility**

- **FR-022**: All existing API endpoints MUST maintain their current request/response schemas.
- **FR-023**: `POST /api/chat` MUST remain accessible without authentication (auth is optional for chat history).
- **FR-024**: `POST /api/translate` and `POST /api/personalize` MUST require JWT authentication.
- **FR-025**: `GET /api/chat/history` MUST require JWT authentication.
- **FR-026**: Error responses (400, 401, 403, 404, 429, 503) MUST use the same detail codes as today.

**Cleanup**

- **FR-027**: The old `services/llm_client.py` (GeminiProvider, GroqProvider, OpenAIProvider, LLMClient failover chain) MUST be removed or replaced.
- **FR-028**: The `groq` Python package dependency MAY be removed if no longer used.
- **FR-029**: The `openai` package MUST remain (it is the SDK underlying `openai-agents`).

### Key Entities

- **Agent**: A configured OpenAI Agents SDK `Agent` object with a name, instructions (system prompt), model reference, and optional tool definitions. Three agents exist: Tutor, Personalization, Translation.
- **Model Configuration**: A shared `AsyncOpenAI` client configured with `base_url` pointing to Gemini's OpenAI-compatible endpoint and `api_key` set to `GOOGLE_API_KEY`.
- **Agent Run**: A single invocation of `Runner.run()` that sends a prompt to the agent, routes through the Gemini endpoint, and returns structured output.

## Success Criteria

### Measurable Outcomes

- **SC-001**: All three agents (Tutor, Personalization, Translation) produce correct outputs when tested with representative inputs — answers are on-topic, personalization adapts to profiles, and translations are in Urdu with code blocks preserved.
- **SC-002**: Full test suite (119+ tests) passes after the migration with no regressions.
- **SC-003**: No Google agentic SDK imports (`google.adk`, `vertexai.agents`, etc.) exist anywhere in the codebase.
- **SC-004**: The `google-genai` SDK is used only in the embeddings path — zero usage in generation.
- **SC-005**: API response times remain comparable to the current implementation (within 20% for equivalent operations).
- **SC-006**: Frontend requires zero changes — all existing fetch calls work without modification.
- **SC-007**: The codebase has fewer total lines in the LLM integration layer compared to the old hand-rolled failover client.
